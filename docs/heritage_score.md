# ReForge Heritage Score

## Purpose

The ReForge Heritage Score is the intelligence engine that determines whether a software project is worth preserving, restoring, and evolving.

Its objective is **not** to measure popularity.

Its objective is to measure **preservation value**.

Every repository receives both:

* A numerical Heritage Score (0–100)
* A detailed Preservation Profile explaining the score

The score is always transparent and explainable.

---

# Core Principle

Popularity does not necessarily imply preservation value.

Likewise, an abandoned project with few stars may contain ideas that are historically important, technically innovative, or educationally valuable.

ReForge evaluates software based on its long-term significance rather than short-term popularity.

---

# Evaluation Framework

## 1. Heritage & Historical Value (25%)

Measures the software's historical significance.

Possible indicators include:

* Influence on other projects
* First-of-its-kind implementations
* Industry adoption
* Academic references
* Historical importance
* Software legacy

Questions:

* Did this software influence an ecosystem?
* Does it represent an important technological milestone?
* Would knowledge be lost if this project disappeared?

---

## 2. Community Value (20%)

Measures continued community interest.

Indicators:

* GitHub Stars
* Forks
* Watchers
* Discussions
* External references
* Downloads (if available)
* Community requests for maintenance

Popularity contributes to preservation decisions but is never the sole deciding factor.

---

## 3. Activity & Sustainability (10%)

Measures the current health of the project.

Indicators:

* Last commit
* Last issue activity
* Pull request activity
* Maintainer availability
* Community maintenance

This metric helps estimate whether restoration is necessary or whether the project is already actively maintained.

---

## 4. Restoration Feasibility (20%)

Measures the likelihood that the project can realistically be restored.

Indicators:

* Build system available
* Dependencies recoverable
* Source completeness
* Test availability
* Documentation quality
* License compatibility

This score estimates engineering effort rather than historical importance.

---

## 5. Educational Value (10%)

Measures how valuable the project is as a learning resource.

Examples include:

* Interesting architecture
* Classic algorithms
* Design patterns
* Game engines
* Compilers
* Frameworks
* Research software

Educational software may deserve preservation even if it has little community activity.

---

## 6. Innovation & Evolution Potential (15%)

Measures whether the project contains ideas worth continuing.

Questions:

* Does it solve a unique problem?
* Are its ideas still relevant?
* Could modern technology significantly improve it?
* Is there clear opportunity for future evolution?

This score evaluates the future potential of the project.

---

# Preservation Profile

Rather than producing only a single score, ReForge generates a multidimensional Preservation Profile.

Example:

Historical Value         ██████████

Community Interest       ████████░░

Educational Value        █████████░

Innovation               ███████░░░

Restoration Feasibility  ██████████

Activity                 ███░░░░░░░

Each category is accompanied by an explanation describing how the score was determined.

---

# Explainability

Every Heritage Score must include a rationale.

Example:

Overall Heritage Score

89 / 100

Reasons:

* Significant historical impact.
* Strong educational value.
* Active community interest despite abandonment.
* Restoration technically feasible.
* Unique architectural ideas worth preserving.

The score should never appear without explanation.

---

# Identity Preservation

ReForge evaluates not only whether software can evolve, but whether its identity can be preserved.

Questions include:

* Can the software be modernized without losing its defining characteristics?
* Will proposed changes preserve the original design philosophy?
* Does evolution strengthen or replace the original project?

Projects with high Identity Preservation are ideal candidates for restoration.

---

# Guiding Question

Before restoration begins, ReForge must answer:

> **Why does this software deserve another chapter?**

If the answer is weak, restoration should not proceed.

---

# Future Improvements

Future versions of the Heritage Score may incorporate:

* Citation analysis
* Dependency graph influence
* Academic usage
* Package ecosystem impact
* Historical snapshots
* Software lineage
* AI-generated significance assessment

---

## Philosophy

The Heritage Score is not a popularity ranking.

It is a preservation framework designed to identify software whose ideas deserve to continue evolving.

Software should not be restored because it is old.

Software should be restored because it is valuable.
